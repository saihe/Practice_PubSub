# 指示履歴 (Instruction History)

このファイルは、本リポジトリに対して与えられた指示と、それに対する設計判断・対応を時系列で記録するものです。
今後新しい指示を受けた場合は、必ずこのファイルの末尾に追記してから実装に着手します（思い出せるようにするため）。

書式:
- **日付**: 指示を受けた日
- **指示**: 受けた内容（要約せず、できるだけ原文の意図を保持）
- **解釈・判断**: 曖昧な点をどう解釈し、どんな既定値・前提を採用したか
- **対応**: 実際に行った実装・成果物

## #1 — 2026-06-06 初回指示（プロジェクト立ち上げ）

### 指示（原文の要旨）
Next + Spring Boot の構成で、Pub/Sub の実装および動きを確認するための Web システムを構築する。確認方法はローカル PC で `docker compose up` を実行し、フロントエンド・バックエンドを立ち上げ、ローカル PC の Web ブラウザ上で確認する。

- Web アプリはタスク管理画面として **1 画面だけ**（タスク一覧を確認する画面）。
- ボタンは 3 つ：**タスク実行（追加）／画面更新／タスク中止**。
- 画面で確認できる項目：**タスクID・追加日時・ステータス・実行時間・処理結果・処理件数・全体件数**。
- タスクドメインのステータス：**キューに追加・開始・正常終了・警告終了・異常終了**。
- タスクが行う処理：
  - キュー時点でランダムな件数を **全体件数** として定義。
  - **1 秒 10 件** くらいでループ。
  - **9 の倍数で失敗** させる。失敗しても中断せず **失敗件数のみカウント**。
  - 失敗件数が 1 件でもあれば **警告終了**。**例外発生時のみ異常終了**。
  - 全体件数は **10〜100 件** の範囲で発生。
- 画面からキュー追加後、何かしらの終了ステータスになったら **スナックバー** で通知。
- 管理画面を開いている間は **デフォルト 10 秒間隔** でフェッチして更新。更新間隔は **5・10・20・30 秒** から選べる。
- タスクの状態管理は **ステータス変更ではなく、ステータスをスタックして最新を取得** する形で実現。
- DB は **H2** で良いが、RDBMS 以外に有力な選択肢があれば提案。
- UI ライブラリは使わず、**Tailwind + ヘッドレス UI** で実装。
- **HATEOAS** を取り入れ、フロントがエンドポイント URL 文字列を知らなくてよいようにする。
- 今回の指示をドキュメントに残す。以降はやり取りなしで実装を完遂する。今後の指示も必ず指示履歴に残す。
- AsciiDoc でも Markdown でもよいので **仕様書を「画面・API・テーブル」でそれぞれ** 作成。
- フロントの実装は **デザインを仕様書として** 残す。コンポーネント単位でデザインを定義し、どの位置にどのコンポーネントを置くか、コンポーネントの状態パターンと状態による変化が分かるようにする。画面仕様にはアクションを定義し、呼び出す API が分かるようにする。

### 解釈・判断（曖昧点の決定）
やり取りなしで完遂する前提のため、以下は当方判断で確定し、根拠を各仕様書にも記載した。

1. **「中止」ステータスの扱い**
   指定 5 ステータスには「中止」が無い。一方で「タスク中止」ボタンには正しい終端状態が必要。
   - 異常終了は「例外発生時のみ」と明記されているため、中止に流用しない。
   - よって **`CANCELLED`（中止終了）を終端ステータスとして追加** した。
   - 指定 5 ステータスは「通常処理の結果」を表し、`CANCELLED` は中止操作専用の拡張、と位置づけた。

2. **正常終了（SUCCESS）の到達性**
   全体件数は 10〜100、かつ 9 の倍数で失敗する。**total ≥ 10 では必ず「9」を通過** するため失敗が 1 件以上発生し、通常運用では **常に警告終了** となる。
   - 仕様に忠実に実装したうえで、この帰結を明記。
   - SUCCESS のコード経路は実装済みで、`app.task.total-count-min` を 9 未満に設定すれば観測可能（既定 10）。

3. **異常終了（ERROR）の観測手段**
   9 の倍数失敗は件数カウントのみ（=警告止まり）で例外ではない。異常終了を観測できるよう、
   **1 タスクあたり確率的に例外を注入**する設定 `app.task.exception-probability`（既定 0.10）を用意。0.0 で無効化できる。

4. **状態のスタック方式**
   `task_status_event` を **append-only**（INSERT のみ、UPDATE なし）とし、最新行（最大 id）を現在状態として読み出す。
   進捗（処理件数）も一定間隔で同テーブルに積むため、ポーリングで進捗が見える。

5. **「中止」ボタンの対象選択**
   ボタンは 3 つという指定に合わせ、一覧で **行を選択（ラジオ/行クリック）** し、選択タスクを中止する方式とした。
   中止可能（未終端）なタスクが選択されている時のみボタンを活性化。

6. **スナックバー通知の対象**
   「画面からキュー追加後」に合わせ、**当セッションで投入したタスク**が終端に達したときに 1 回だけ通知する。

7. **HATEOAS の徹底**
   フロントがハードコードする URL は **API ルート `http://localhost:8080/api` のみ**。
   一覧・作成・中止・履歴は、サーバが返す `_links` を辿って発見する。

8. **DB 選定（提案）**
   デモのゼロ依存性・起動の速さを優先し **H2（インメモリ）** を採用。
   RDBMS 以外の有力候補（Redis Streams 等）と比較検討した結果は `docs/architecture.md` に記載。

9. **Pub/Sub の実装方式**
   外部ブローカを増やさずに「発行/購読」を体験できるよう、Spring の `ApplicationEventPublisher` /
   `@EventListener` + `@Async` を用いた **アプリ内 Pub/Sub** を採用。トピック型ブローカへの対応関係も
   `docs/architecture.md` に記載。

### 対応（成果物）
- バックエンド: Spring Boot 3.3 / Java 21 / Spring Web / Spring HATEOAS / Spring Data JPA / H2。
  - Pub/Sub（発行→購読の非同期処理）、append-only ステータススタック、協調的キャンセル、HATEOAS。
  - テスト 5 件（処理ルール・中止・HATEOAS リンク）すべて green。
- フロントエンド: Next.js 14 / TypeScript / Tailwind / Headless UI。
  - 1 画面、3 ボタン、間隔セレクト（5/10/20/30 秒・既定 10）、スナックバー、HATEOAS クライアント。
- Docker: フロント/バックそれぞれ Dockerfile、ルートに `docker-compose.yml`。
- ドキュメント: 本ファイル、`architecture.md`、`spec-screen.md`、`spec-api.md`、`spec-table.md`、`spec-design.md`。
- 動作確認: ローカルでバックエンド（jar）+ フロント（next）を起動し、ブラウザで
  追加→進捗→警告終了→スナックバー、および中止→中止終了 までを実機確認済み。

## #2 — 2026-06-06 Docker Desktop が WSL 関連で起動できない問題の解消

### 指示（原文の要旨）
PC にインストールした Docker Desktop が WSL 関連で起動できない。Docker 公式の Windows インストール手順（system requirements のリンク）を確認し、この PC で Docker Desktop を使用できるようにする。

### 診断結果（このPCの実状態）
- OS: Windows 11 Pro 26200（要件 22631 以上を満たす）、**arch は arm64**。
- WSL 本体: インストール済み（`wsl --version` = 2.7.3.0、既定バージョン 2）。要件 2.1.5 以上を満たす。
- `wsl --status`: 「このコンピューターで仮想化が有効になっていないため、WSL2 を開始できません。[仮想マシン プラットフォーム] オプション コンポーネントが有効になっており、ファームウェア設定で仮想化がオンになっていることを確認してください」。
- `wsl --list --verbose`: **ディストリビューションが 1 つも入っていない**。
- `systeminfo`: 「A hypervisor has been detected」「Virtualization-based security: Running」＝ VBS/メモリ整合性により Hyper-V ハイパーバイザーは稼働中（＝CPU 仮想化自体は有効）。

### 解釈・判断
- systeminfo でハイパーバイザーが検出されている＝ファームウェアの仮想化は ON。よって BIOS 変更は不要。
- 矛盾するように見える WSL の「仮想化が無効」メッセージの実体は、**Windows 機能「仮想マシン プラットフォーム（VirtualMachinePlatform）」が未有効**であること。これが Docker Desktop の WSL2 バックエンド起動を妨げている根本原因と判断。
- 対処は公式手順どおり `wsl --install --no-distribution`（WSL + 仮想マシン プラットフォーム機能を有効化）。**管理者権限が必須**のため UAC 昇格を伴う。Docker Desktop 自体は専用 distro（docker-desktop）を自前生成するため、ユーザー distro の追加導入は必須ではない。
- 機能有効化後は **再起動が必須**。再起動後に Docker Desktop を起動して動作確認する。

### 対応
- 本診断と手順を記録。`wsl --install --no-distribution` を管理者権限（UAC 承認）で実行し、**仮想マシン プラットフォーム機能の有効化に成功**（`HKLM\...\Component Based Servicing\RebootPending` = True で再起動待ちを確認）。
- **再起動が必要**な状態でここまで完了。残作業: PC 再起動 → Docker Desktop 起動 → 設定で「Use the WSL 2 based engine」ON を確認 → `docker version`（サーバー側）/`wsl --status` で正常化を検証。再起動はユーザー作業の都合があるため自動実行せずユーザーに委ねた。

## #3 — 2026-06-07 終了通知をリアルタイム化（ポーリング依存の解消／SSE 導入）

### 指示（原文の要旨）
「だめかも。何かしらの終了の通知が、画面更新で新しいステータスが取得されたときになってて、リアルタイムになってない。」

→ タスクが終端に達したときのスナックバー通知が、自動更新（ポーリング）のフェッチ時にしか発火せず、最大で「更新間隔（既定10秒）」ぶん遅れる。これを**リアルタイム**にしてほしい、という指摘。

### 原因（調査結果）
- 終了検知 `detectTerminals()` は `loadTasks()`（=ポーリング or 画面更新）の中でのみ呼ばれていた（`frontend/src/app/page.tsx`）。
- そのためバックエンドで終端ステータスが積まれても、フロントは次のフェッチまで気づけず、通知が遅延する構造だった。

### 解釈・判断
やり取りなしで完遂する前提のもと、以下で確定した。

1. **配信方式は SSE（Server-Sent Events）を採用。**
   既存の Pub/Sub テーマ（`ApplicationEventPublisher` + `@EventListener`）に最も自然に乗る。新たな外部ブローカや WebSocket 双方向通信は不要で、サーバ→ブラウザの一方向 push で十分。**「購読側（サブスクライバ）をブラウザまで延長する」**という位置づけで、1メッセージが複数ブラウザへファンアウトする Pub/Sub の特徴もそのまま体験できる。
2. **配信対象は終端だけでなく全ステータス変更（進捗含む）。**
   `TaskStatusChangedEvent` は STARTED・進捗・終端のすべてで発行済み。これを購読して push することで、**通知だけでなく一覧の進捗もリアルタイム化**できる（最小変更で最大効果）。
3. **ポーリングは廃止せず「フォールバック（再同期）」として残す。**
   SSE 切断時や取りこぼしの保険。間隔セレクト（5/10/20/30秒）も従来どおり機能。SSE が主、ポーリングが従。
4. **HATEOAS を維持。**
   SSE エンドポイント `GET /api/tasks/stream` を一覧コレクションの `_links.stream` として公開し、フロントは URL 文字列をハードコードせずリンクを辿って購読する。
5. **SSE ペイロードは `TaskView`（現在状態に畳み込んだスナップショット）。**
   既存の読み取りモデルを再利用。REST と同じ Jackson 設定でシリアライズされるため日時等の形式も一致。フロントは `taskId` で既存行へマージし、`_links` は手元の値を温存（終端時は `cancel` リンクを除去）。

### 対応（成果物）
- バックエンド:
  - `pubsub/TaskStreamSubscriber.java` を新規追加。`TaskStatusChangedEvent` を購読し、接続中の各 `SseEmitter` へ対象タスクの最新 `TaskView` を push。接続/タイムアウト/エラーで自動的に購読解除。
  - `web/TaskController.java` に `GET /api/tasks/stream`（`text/event-stream`）を追加し、一覧コレクションに `rel=stream` リンクを付与。
- フロントエンド（`frontend/src/app/page.tsx`, `lib/api.ts`, `lib/types.ts`）:
  - 一覧の `_links.stream` から `EventSource` を開き、`task` イベントを受信したら該当行をマージ＆その場で `detectTerminals()` を実行 → **終端到達と同時にスナックバー通知**。進捗・最終更新時刻もリアルタイム反映。
  - ポーリングはフォールバックとして継続。UI 文言を「リアルタイム（SSE）＋フォールバック更新」に微修正。
- 動作確認: バックエンド＋フロントをローカル起動し、タスク追加→進捗が秒単位で更新→終端と同時にスナックバー通知が出ることを確認（ポーリング間隔に依存しない）。

## #4 — 2026-06-07 フォールバック方針の見直し（SSE主体＋Last-Event-IDで取りこぼしゼロ）

### 指示（原文の要旨）
「SSEでリアルタイム反映できるなら、フォールバックは要らないのでは？ 議論したい。SSEのみにするか、フォールバックで画面反映＋終了時通知にするか。クライアントや通信に負荷がないならリアルタイムでいい気がしている。」

### 議論・解釈
判断軸は「負荷」ではなく「**SSEが切れたときに何を取りこぼすか**」と整理した。
- EventSource は自動再接続するが、**現状は各イベントに `id` を振っておらず、再接続時の `Last-Event-ID` 再送もしていない**ため、切断中に積まれたイベント（＝終端＝終了通知）を取りこぼす。これは利用者が最も気にしている当の対象。
- 状態は append-only スタックの「最新1件への畳み込み」なので、**再接続時に1回フェッチし直せば完全に整合**する＝常時ポーリングは過剰。
- 自動更新間隔セレクト（5/10/20/30秒）は指示#1の明示要件のため、撤廃せず「切断時の再取得間隔」として意味を持たせる。手動「画面更新」も維持。

選択肢として A（SSE主体＋切断時のみ再同期）／B（SSEのみ）／C（現状維持＝SSE＋定期ポーリング）／A＋Last-Event-ID を提示。**利用者は「A＋Last-Event-IDで取りこぼしゼロ」を選択。**

### 対応（成果物）
- **バックエンド（SSEを at-least-once 相当に）**:
  - `TaskStatusEvent` の行ID（グローバル単調増加）を SSE の `id:` として配信。`TaskStatusChangedEvent` に `eventId` を追加し、`TaskEventStore.append` で採番済み行IDを載せて publish。
  - `GET /api/tasks/stream` で `Last-Event-ID` ヘッダを受け、`TaskStreamSubscriber.subscribe(lastEventId)` が**それ以降の未受信イベントを再送（リプレイ）してから live 配信へ合流**。リプレイはリクエストスレッドのみで実施し、同一 emitter への並行書き込みを回避。
  - 追加: `TaskStatusEventRepository.findByIdGreaterThanOrderByIdAsc`、`TaskService.statusEventsAfter`。
- **フロントエンド（Option A 挙動）**:
  - **SSE が健全な間は定期ポーリングを停止**（`live` のとき interval を張らない）。**切断中だけ**「切断時の再取得間隔」でポーリング。
  - **接続/再接続のたびに1回 reconcile フェッチ**（`connected` 受信時に `loadTasks(false)`）。Last-Event-ID 再送の隙間の backstop。
  - 文言調整: 接続中は「自動更新: SSE リアルタイム（ポーリング停止）」、未接続時のみ「N 秒ごとに再取得」。間隔ラベルを「切断時の再取得間隔」へ。
- 動作確認: 8081 のローカル機で、(1)通常接続でリアルタイム反映＋停止中ポーリング、(2)バックエンド再起動による切断→再接続で `Last-Event-ID` リプレイ＋reconcile により取りこぼしなく追従、をブラウザ実機で確認。バックエンド `mvn clean test` 全グリーン、フロント `tsc --noEmit` クリーン。
