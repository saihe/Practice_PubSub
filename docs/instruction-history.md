## #1 — 2026-06-06 初回指示（プロジェクト立ち上げ）

### 指示（原文の要旨）
Next + Spring Boot の構成で、Pub/Sub の実装および動きを確認するための Web システムを構築する。確認方法はローカル PC で `docker compose up` を実行し、フロントエンド・バックエンドを立ち上げ、ローカル PC の Web ブラウザ上で確認する。

- Web アプリはタスク管理画面として **1 画面だけ**（タスク一覧を確認する画面）。
- ボタンは 3 つ：**タスク実行（追加）／画面更新／タスク中止**。
- 画面で確認できる項目：**タスクID・追加日時・ステータス・実行時間・処理結果・処理件数・全体件数**。
- タスクドメインのステータス：**キューに追加・開始・正常終了・警告終了・異常終了**。
- タスクが行う処理：
  - キュー時点でランダムな件数を **全体件数** として定義。
  - **1 秒 10 件** くらいでループ。
  - **9 の倍数で失敗** させる。失敗しても中断せず **失敗件数のみカウント**。
  - 失敗件数が 1 件でもあれば **警告終了**。**例外発生時のみ異常終了**。
  - 全体件数は **10〜100 件** の範囲で発生。
- 画面からキュー追加後、何かしらの終了ステータスになったら **スナックバー** で通知。
- 管理画面を開いている間は **デフォルト 10 秒間隔** でフェッチして更新。更新間隔は **5・10・20・30 秒** から選べる。
- タスクの状態管理は **ステータス変更ではなく、ステータスをスタックして最新を取得** する形で実現。
- DB は **H2** で良いが、RDBMS 以外に有力な選択肢があれば提案。
- UI ライブラリは使わず、**Tailwind + ヘッドレス UI** で実装。
- **HATEOAS** を取り入れ、フロントがエンドポイント URL 文字列を知らなくてよいようにする。
- 今回の指示をドキュメントに残す。以降はやり取りなしで実装を完遂する。今後の指示も必ず指示履歴に残す。
- AsciiDoc でも Markdown でもよいので **仕様書を「画面・API・テーブル」でそれぞれ** 作成。
- フロントの実装は **デザインを仕様書として** 残す。コンポーネント単位でデザインを定義し、どの位置にどのコンポーネントを置くか、コンポーネントの状態パターンと状態による変化が分かるようにする。画面仕様にはアクションを定義し、呼び出す API が分かるようにする。

### 解釈・判断（曖昧点の決定）
やり取りなしで完遂する前提のため、以下は当方判断で確定し、根拠を各仕様書にも記載した。

1. **「中止」ステータスの扱い**
   指定 5 ステータスには「中止」が無い。一方で「タスク中止」ボタンには正しい終端状態が必要。
   - 異常終了は「例外発生時のみ」と明記されているため、中止に流用しない。
   - よって **`CANCELLED`（中止終了）を終端ステータスとして追加** した。
   - 指定 5 ステータスは「通常処理の結果」を表し、`CANCELLED` は中止操作専用の拡張、と位置づけた。

2. **正常終了（SUCCESS）の到達性**
   全体件数は 10〜100、かつ 9 の倍数で失敗する。**total ≥ 10 では必ず「9」を通過** するため失敗が 1 件以上発生し、通常運用では **常に警告終了** となる。
   - 仕様に忠実に実装したうえで、この帰結を明記。
   - SUCCESS のコード経路は実装済みで、`app.task.total-count-min` を 9 未満に設定すれば観測可能（既定 10）。

3. **異常終了（ERROR）の観測手段**
   9 の倍数失敗は件数カウントのみ（=警告止まり）で例外ではない。異常終了を観測できるよう、
   **1 タスクあたり確率的に例外を注入**する設定 `app.task.exception-probability`（既定 0.10）を用意。0.0 で無効化できる。

4. **状態のスタック方式**
   `task_status_event` を **append-only**（INSERT のみ、UPDATE なし）とし、最新行（最大 id）を現在状態として読み出す。
   進捗（処理件数）も一定間隔で同テーブルに積むため、ポーリングで進捗が見える。

5. **「中止」ボタンの対象選択**
   ボタンは 3 つという指定に合わせ、一覧で **行を選択（ラジオ/行クリック）** し、選択タスクを中止する方式とした。
   中止可能（未終端）なタスクが選択されている時のみボタンを活性化。

6. **スナックバー通知の対象**
   「画面からキュー追加後」に合わせ、**当セッションで投入したタスク**が終端に達したときに 1 回だけ通知する。

7. **HATEOAS の徹底**
   フロントがハードコードする URL は **API ルート `http://localhost:8080/api` のみ**。
   一覧・作成・中止・履歴は、サーバが返す `_links` を辿って発見する。

8. **DB 選定（提案）**
   デモのゼロ依存性・起動の速さを優先し **H2（インメモリ）** を採用。
   RDBMS 以外の有力候補（Redis Streams 等）と比較検討した結果は `docs/architecture.md` に記載。

9. **Pub/Sub の実装方式**
   外部ブローカを増やさずに「発行/購読」を体験できるよう、Spring の `ApplicationEventPublisher` /
   `@EventListener` + `@Async` を用いた **アプリ内 Pub/Sub** を採用。トピック型ブローカへの対応関係も
   `docs/architecture.md` に記載。

### 対応（成果物）
- バックエンド: Spring Boot 3.3 / Java 21 / Spring Web / Spring HATEOAS / Spring Data JPA / H2。
  - Pub/Sub（発行→購読の非同期処理）、append-only ステータススタック、協調的キャンセル、HATEOAS。
  - テスト 5 件（処理ルール・中止・HATEOAS リンク）すべて green。
- フロントエンド: Next.js 14 / TypeScript / Tailwind / Headless UI。
  - 1 画面、3 ボタン、間隔セレクト（5/10/20/30 秒・既定 10）、スナックバー、HATEOAS クライアント。
- Docker: フロント/バックそれぞれ Dockerfile、ルートに `docker-compose.yml`。
- ドキュメント: 本ファイル、`architecture.md`、`spec-screen.md`、`spec-api.md`、`spec-table.md`、`spec-design.md`。
- 動作確認: ローカルでバックエンド（jar）+ フロント（next）を起動し、ブラウザで
  追加→進捗→警告終了→スナックバー、および中止→中止終了 までを実機確認済み。
